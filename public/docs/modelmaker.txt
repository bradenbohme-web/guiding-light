Step 1 — ReferencePack + Ortho Overlay System
1) Goal and invariants

You want the reference images to behave like CAD underlays:

1 unit = 1 meter (world space)

orthographic cameras (no perspective)

each view image is placed on a plane whose size in meters is correct using the 1m scale bar

each view is aligned by centerline + two anchor points (e.g., bow tip + transom, or front bumper + rear bumper)

the system is deterministic and saved as data (so you can reopen and continue tuning later)

Once this is done, everything else (curve extraction, silhouette error, auto-fitting) becomes straightforward.

Store everything needed to align without guessing.

Meaning (no ambiguity)

pxPerMeter: measured from the 1m bar in that view

originPx: pixel that maps to world origin in that view (usually center of hull at some datum)

centerlinePx: two pixel points defining the centerline direction (to compute rotation)

anchorA_Px, anchorB_Px: two pixels on the centerline that define longitudinal positioning (bow/stern, front/rear)

opacity: how strong the underlay is in the viewport

We never trust AI-printed dimensions. Only pxPerMeter from the bar + your anchors.

3) View conventions (universal)

You must standardize how each “view plane” maps into world meters.

Assume your asset frame is:

X = forward

Y = up

Z = right

Then each view has a fixed mapping:

TOP view plane

image +X points to world +X

image +Y points to world -Z (because image y increases downward)

plane sits at some constant Y (above object)

SIDE view plane (port/starboard)

image +X → world +X

image +Y → world -Y

plane sits at some constant Z (beside object)

FRONT view plane

image +X → world +Z

image +Y → world -Y

plane sits at some constant X (in front of object)

REAR view plane

image +X → world -Z (mirrored to face camera cleanly)

image +Y → world -Y

plane sits at some constant X (behind object)

This mapping is the backbone. Everything else is just scale + translation + rotation.

4) Calibration: turn pixels into meters

Given an image of size (Wpx, Hpx) and pxPerMeter:

metersPerPx = 1 / pxPerMeter

plane size in meters:

Wm = Wpx * metersPerPx

Hm = Hpx * metersPerPx

Pixel-to-plane coordinates (meters) with origin at originPx = (ox, oy):

u = (px - ox) * metersPerPx

v = (py - oy) * metersPerPx

(Here v is “down is positive” in image coordinates; we’ll flip when mapping to world.)

5) Alignment: rotation from the centerline

From centerlinePx = [p0, p1]:

dirPx = normalize(p1 - p0)

compute its angle in image space: θ = atan2(dirPx.y, dirPx.x)

For TOP and SIDE views, the desired direction is “perfect horizontal” along +X, so you rotate the plane (or equivalently rotate the texture UV transform) by -θ.

Then you translate so that:

anchorA_Px maps to your chosen world anchor (e.g., bow tip at x = +L/2)

anchorB_Px maps to the stern plane (x = -L/2)

If you don’t know world L yet, you can simply align anchors to your current model bow/stern points (interactive), and later the fit system will tune L.

6) Three.js implementation pattern (works in vanilla Three or R3F)
A) A “ReferencePlane” object per view

Each is:

a plane mesh with the image texture

a fixed orthographic camera matched to that view

user controls: opacity, show/hide, nudge, lock

B) The plane placement

You can place each plane at a constant offset so it never intersects geometry:

TOP plane at Y = +Ymax + margin

SIDE plane at Z = -(Zmax + margin) (or +)

FRONT plane at X = +Xmax + margin

REAR plane at X = -(Xmax + margin)

The plane should always face the camera.

7) Concrete code skeleton (vanilla Three.js math)

This is the core transform logic you’ll reuse everywhere.

type Px = [number, number];
type ViewAxis = "TOP" | "SIDE" | "FRONT" | "REAR";

interface ViewCal {
  pxPerMeter: number;
  originPx: Px;
  centerlinePx: [Px, Px];
  anchorA_Px: Px;
  anchorB_Px: Px;
}

interface ViewDef {
  imageUrl: string;
  axis: ViewAxis;
  cal: ViewCal;
  opacity: number;
}

function pxToMeters(px: Px, cal: ViewCal): { u: number; v: number } {
  const mpp = 1 / cal.pxPerMeter;
  const u = (px[0] - cal.originPx[0]) * mpp;
  const v = (px[1] - cal.originPx[1]) * mpp;
  return { u, v };
}

function centerlineAngle(cal: ViewCal): number {
  const [a, b] = cal.centerlinePx;
  const dx = b[0] - a[0];
  const dy = b[1] - a[1];
  return Math.atan2(dy, dx); // image-space angle
}

/**
 * Returns a local-plane transform that:
 * 1) scales plane to meters
 * 2) rotates to make centerline horizontal
 * 3) positions so that originPx is at world origin (you can add extra offsets per view)
 */
function buildReferencePlaneTransform(view: ViewDef) {
  const theta = centerlineAngle(view.cal);

  // rotate by -theta so centerline becomes horizontal
  const rotZ = -theta;

  // originPx maps to local plane origin by construction, so translation is zero here.
  // We'll set final world translation by placing the plane at a fixed offset (per view).
  return { rotZ };
}


This is intentionally minimal. The final world orientation depends on axis mapping (next section).

8) Axis mapping: plane local (u,v) → world

After you’ve built a plane where:

local +u is “right”

local +v is “down” (image convention)

You map into world per view:

TOP

worldX = +u

worldZ = -v

worldY = constant

SIDE

worldX = +u

worldY = -v

worldZ = constant

FRONT

worldZ = +u

worldY = -v

worldX = constant

REAR

worldZ = -u (mirror)

worldY = -v

worldX = constant

This mapping is what makes the same pack schema work for boats and cars.

10) Ortho camera setup (so your renders match underlays)

Each view camera must be orthographic and match the plane’s meters.

Use one rule:

set camera frustum so that the visible width/height in world meters matches your plane coverage.

Example: if TOP plane is Wm x Hm, set:

left = -Wm/2, right = +Wm/2

top = +Hm/2, bottom = -Hm/2

Then position the camera above, looking down:

TOP: camera at (0, +d, 0) looking at origin

SIDE: camera at (0, 0, +d) looking at origin

FRONT: camera at (+d, 0, 0) looking at origin

REAR: camera at (-d, 0, 0) looking at origin

Now your model render and underlay are in the same metric space.

Step 2 — Curve-First Generator (the universal “base form” engine)

The goal of Step 2 is simple and absolute:

Every “main shape” asset (hull, car body, fuselage, fairing) is driven primarily by 3 longitudinal curves — and everything else is derived or layered as features later.

Those three curves are:

B(u) = half-beam / half-width distribution along length

K(u) = keel / underside profile (rocker for boats, floor line for cars, belly line for aircraft)

D(u) = deck / topside profile (sheer line, roofline, upper silhouette)

Where u ∈ [0,1] is normalized length.

Once you have this, you can fit to ortho packs, keep the whole thing editable forever, and scale to any object class.

2.1 Canonical parameterization
Asset axis conventions

World X = forward (length)

World Y = up (height)

World Z = right (beam)

Normalized coordinate

u = 0 stern, u = 1 bow (or vice versa — choose once and never change)

Convert to meters:

x(u) = (u - u0) * L
Typically u0 = 0.5 so x ∈ [-L/2, +L/2]

Cross-section coordinate

For a given u, we define a normalized lateral coordinate:

actual half-beam: b = B(u) (meters)

lateral position: z ∈ [-b, +b]

normalized lateral magnitude: s = clamp(|z| / max(b, ε), 0, 1)

So s=0 is centerline (keel), s=1 is rail / max beam.

For boats and cars, most “identity” is in planform.

B(u) should be a spline you can fit to top ortho silhouette.

Strong recommendation

2.6 Mesh synthesis: build a parametric surface grid

Stop thinking “deform a box” as the concept.
Under the hood, do this:

2.6.1 Build a half-surface chart

Indices:

i = 0..Nu along u

j = 0..Nv across s (0..1)

For each (i,j):

u = i/Nu

s = j/Nv

b = B(u)

choose actual z:

z = s * b (half hull, z ≥ 0)

Compute:

y = yK + (yD - yK) * F(s)

x = x(u)

So the half surface point is:

P = (x, y, z)

Then mirror to get the other half:

P' = (x, y, -z)

u (0..1)

2.11 The generator algorithm (universal)

High-level:

sample u stations

for each station, compute b, yK, yD, sectionParams

for each lateral sample, compute s, then t = F(s)

compute vertex position

mirror for other side

build indices

compute normals

That’s it. Everything else is features and refinements.

Step 3 — Feature Node System (the universal “add variables forever” engine)

Step 2 gives you a perfect, fit-able base shell.
Step 3 is how you add everything else—cockpit cutouts, centerboard trunks, fender arches, hood scoops, antenna bosses, mirror curvature tweaks—without corrupting the base.

The target is a feature graph where each feature:

3.1 Feature Graph Model
3.1.1 Node graph vs pipeline

Internally, you compile the graph into a linear pipeline (ordered passes), because:

deterministic

fast

composable

3.3 The 5 Core Feature Types

If you implement these five node types cleanly, you can build almost anything.

Feature Type A — PatchDisplace (bulge/scoop/flare)

Operation: push vertices along surface normal (or a preferred direction).

Params:

center (u0,s0)

radiusU, radiusS

height (+/-)

profile (bell, ridge, plateau)

edgeSharpness

Output:

Δp = n * height * profile(r) * m

Uses:

hood scoop bump

fender flare

deck crown enhancement

bow shoulder fattening

Feature Type B — RecessCut (cockpit/vent/well)

Operation: push inward (negative normal) + optional rim shaping.

You do not do triangle booleans unless needed.
You “carve” using a recess field + rim lift.

Params:

shape mask (rounded rect, curve loop, etc.)

depth

rimWidth

rimHeight

rimRoundness

Output:

interior: Δp = -n * depth * m

rim: add back a lip with a band-pass mask:

rimMask = band(m, inner=0.7, outer=0.95) (example)

Δp += n * rimHeight * rimMask

Uses:

Laser cockpit subtraction

centerboard trunk recess area

car window cut impressions

vent cutouts (visual)

If you need a real hole: you do it as a controlled topology submesh later (Step 6 export/boolean insert), not by fragile booleans on random triangles.

Feature Type C — CreaseLine (chines/panel breaks)

Operation: locally pinch normals / displace around a curve.

Params:

curve C(u) or spline in (u,s) or on surface

width

sharpness

depth (optional)

Implementation:

d = distance_to_curve(vertex)

m = smoothstep(width, 0, d)^sharpness

displace along a directional field (often normal, sometimes tangent/binormal) to create “break”

Uses:

hull chine

Laser deck/hull seam emphasis

car door panel lines (macro)

aircraft fairing edges

This is huge because “edge language” gives identity without topology violence.

Feature Type D — VectorWarp (mirror shaping, twist, local sculpt)

Operation: warp vertices by a vector field, not necessarily normal.

Params:

mask (ellipse/curve distance)

warp direction:

along +X/+Y/+Z

or local tangent/bitangent

or custom “flow field”

amplitude profile

Output:

Δp = field(p) * amplitude * m

Uses:

side mirror backside curve correction

bow tip alignment without altering beam curve globally

subtle asymmetries (if desired)

helicopter canopy shaping

Feature Type E — BossHole (mounts, rivets, antenna bases)

Operation: build a bump (boss) + optional hole depression.

For stability, you do it as:

boss = PatchDisplace with small ellipse + “flat-top” profile

hole = RecessCut with circular mask

optional: add a hardpoint

Params:

location (u,s) or anchor

boss radius/height

hole radius/depth

fillet feel

Uses:

Laser rigging attachments

rudder gudgeon pads

car antenna mount

bolt heads / fasteners (macro-level)

3.4 Feature Staging (critical order)

Your pipeline stages should be fixed and global:

Base Form (Step 2)

Macro Features (cockpit recess, wheel wells, trunk)

Edge Language (creases, seam sharpness, lip shaping)

Micro Features (bosses, small mounts)

Finish (normals, bevel approximations, decal seams)

Why this matters:

if you crease before you cut, your rim gets destroyed

if you add bosses before a macro warp, they drift

So stages are a hard constraint.

Step 4 — Metrics + Fitting Loop (make ortho packs become “geometry unit tests”)

This step turns your system from “editable” into self-correcting.

You will build a loop that can:

render your model in true orthographic views (top/side/front/rear)

extract silhouettes + keypoints from those renders

compare them against the calibrated ReferencePack underlays

compute error curves in meters

propose and apply a bounded parameter patch (human or LLM)

repeat until error converges

Everything that follows (cars, helicopters, buildings) becomes the same pipeline.

4.1 The fitting philosophy (absolute and universal)
Hard constraints

Ortho cameras are ground truth.

ReferencePack scale bars define meters.

You optimize only within your typed action space:

spline control points for B(u), K(u), D(u)

section-style params (bow/mid/stern stations)

feature node params (local edits)

enable/disable nodes

Trust hierarchy (prevents drift)

Top view dominates width → fit B(u) primarily from TOP

Side view dominates height → fit K(u) and D(u) primarily from SIDE

Front/Rear dominate section style → fit SectionParams(u) primarily from FRONT/REAR
If conflicts exist, you don’t average blindly—you obey the hierarchy.

4.2 OrthoRenderPack (your model → images in metric space)

You already built Step 1 alignment. Now you need a standardized render output:

For each view V ∈ {top, side, front, rear} render into an offscreen target:

resolution: start at 1024×1024 (plenty; 512 also works)

flat shader:

hull/body = solid white

background = transparent

disable lighting, textures, post-processing

output should contain only silhouette geometry (optionally include separate pass for deck vs hull)

Why: silhouette extraction is easiest from a binary alpha mask.

4.3 Silhouette extraction (fast, robust, works in Three.js)
4.3.1 Readback

renderer.readRenderTargetPixels(rt, 0, 0, W, H, buffer)

threshold alpha: a > 10 (or similar)

4.3.2 Convert pixels → meters (per view)

Given your ortho camera frustum:

width in meters: Wm = right - left

height in meters: Hm = top - bottom

Then:

metersPerPixelX = Wm / Wpx

metersPerPixelY = Hm / Hpx

And you can map pixel coordinates into world-plane meters directly.

4.3.3 Extract 1D silhouette functions (the key trick)

Don’t store all edge pixels. Store scanline extrema to get clean curves.

TOP view (width distribution):
For each x_px (or each column), find:

z_left(x) = smallest y_px with solid? (depends orientation)

z_right(x) = largest y_px with solid
Convert to meters:

halfBeam_model(x) = 0.5 * (z_right - z_left)

SIDE view (rocker + sheer):
For each x column:

y_bottom(x) = lowest solid pixel

y_top(x) = highest solid pixel
Convert to meters:

K_model(x) = y_bottom(x)

D_model(x) = y_top(x)

FRONT/REAR (section style):
For each y row:

z_left(y), z_right(y) → width vs height profile
This provides section targets (not necessarily full reconstruction).

You now have compact, smooth-ish arrays:

B_model[i]

K_model[i]

D_model[i]

Store them sampled at N stations (e.g., 256).

4.4 Reference silhouette extraction (from ReferencePack)

You do the same scanline extrema extraction on the reference images.

But first:

you must apply the ReferencePack calibration:

pxPerMeter

origin alignment

centerline rotation correction (if any)

Practical: you can precompute and cache:

B_ref(x), K_ref(x), D_ref(x)

plus keypoints

4.5 Error metrics (your objective function)

You want both “global score” and “diagnostic curves.”

4.5.1 Signed error curves (most important)

At each sampled station i along length:

eB[i] = B_model[i] - B_ref[i] (meters)

eK[i] = K_model[i] - K_ref[i]

eD[i] = D_model[i] - D_ref[i]

These three curves tell you exactly where you’re off.

4.5.2 Scalar scores (for convergence and automation)

RMS:

rmsB = sqrt(mean(eB^2))

rmsK, rmsD

Max absolute error:

maxB = max(|eB|) etc.

Area mismatch (optional):

difference in silhouette area in each view

4.5.3 Fairness penalties (prevents “wiggly cheating”)

You must discourage fitting by adding noise.

For each curve error or for your spline itself:

discrete second-derivative energy:

E_fair(B) = Σ (B[i+1] - 2B[i] + B[i-1])^2

4.6 Keypoints (for locking the “identity”)

Silhouette curves alone can drift globally (e.g., whole hull shifts).

So you define a small set of keypoints per view.

Examples:

Boat

bow tip (max X extent in top/side)

transom plane corners (rear view)

max beam station (argmax of B_ref)

cockpit corners (if visible)

mast step (if drawn)

Car

axle centers (critical)

wheel arch peak points

roof highest point

bumper endpoints

Keypoints can be:

auto-detected from extrema

or user-clicked once and saved into ReferencePack

Keypoint errors become extra terms:

eKP = ||p_model - p_ref||

Keypoints stabilize scale/placement while curves handle shape.

4.7 Optimization strategy (stable, fast, and universal)

You do multi-scale fitting:

Stage A — Fit base curves only (big shape)

Variables:

spline control points for B(u), K(u), D(u)

maybe global L if you allow it

Objective:

minimize eB, eK, eD with fairness

Algorithm options:

Least-squares on spline control points (best):

treat curve as spline with small number of controls (8–16)

solve with regularization (fairness)

Coordinate descent (simple fallback):

nudge one control point at a time

keep if improves score

Finite-difference Gauss-Newton (more aggressive):

small dims → feasible

Start with (1). It gives professional smooth results.

Stage B — Fit section style (cross-section cues)

Variables:

SectionParams at a few stations (bow/mid/stern), interpolated

Objective:

match front/rear profiles (width vs height), plus subjective constraints (V vs round)

Stage C — Fit macro features (localized residual)

Enable nodes like cockpit recess / trunk / wheel wells.
Variables:

feature node params only

Objective:

reduce localized silhouette mismatch (in a mask region)

keep global curves mostly locked

Stage D — Final polish

Small adjustments across all, but bounded.

This is how you avoid “feature fighting” and keep convergence monotonic.

5.2 LOD Generation (do it by design, not decimation)

Decimation is fragile for parametric assets. You already control topology through sampling density, so LOD is “just rebuild with fewer samples.”

5.2.1 LOD presets (universal)

Define LODs by (Nu, Nv) plus feature toggles.

Example:

LOD0: Nu=256, Nv=96 (high)

LOD1: Nu=160, Nv=64

LOD2: Nu=96, Nv=40

LOD3: Nu=64, Nv=24 (optional)

5.2.2 Feature-aware simplification

For lower LODs:

disable micro features (bosses, tiny lips)

keep macro silhouette features (cockpit, wheel arches)

preserve seams/creases (cheap identity)

This is controlled by feature node “importance”:

importance: "macro" | "edge" | "micro"

So the compiler can automatically drop micro features for LOD2+.

5.2.3 “Silhouette protection”

You can optionally enforce:

LOD1/2 must keep top and side silhouette within ε meters of LOD0.
If not, increase samples or keep a macro feature.

This becomes an internal quality gate.

5.3 Collision Output (three tiers)

Pick based on gameplay needs.

Tier A — Primitive colliders (fastest)

boxes/capsules/spheres aligned to anchor frames

best for vehicles/boats in arcade physics

Tier B — Simplified collision mesh (balanced)

generate a low-resolution hull/shape mesh directly from LOD2

optionally shrink-wrap to avoid jitter

best for stable contact and interior cutouts

Tier C — Hybrid (recommended universal default)

macro collision mesh + primitive supplements

e.g., boat hull mesh + capsule for mast base zone, etc.

Key rule: collision is a separate build target. Never “reuse LOD0” as collision.

5.4 Field Proxies (the cheap “SDF benefits” you actually want)

  ]
}


Generate them deterministically:

sample stations along u

at each station place 1–3 spheres across beam

radius tied to local beam and depth

5.4.2 Capsule chains (good for long structures)

Mast, boom, fuselage: capsules are perfect.

5.4.3 Aero proxies (optional)

For aircraft/helis: store surface patches or spanwise sample points with normals/areas.

5.5 Hardpoints (attachments as first-class data)

5.8 LLM Patch Protocol (safe, bounded, convergent)

This is how you actually leverage multimodal models without chaos.

5.8.1 Inputs you give the model

5.8.2 Output you accept

rms_total decreases by ≥ δ

fairness penalty does not increase past threshold

no keypoint errors worsen beyond tolerance
Otherwise reject (or ask for alternate patch).

This makes the LLM an optimizer under supervision, not an author.

5) WheelGraphPreset (32 spokes + 3-cross) — the real unlock

5.1 Wheel parameters

N = 32 spokes

R_rim rim radius

R_hub hub flange radius

flangeOffsetY half distance between hub flanges (left/right)

cross = 0|1|2|3

holesRim = N, holesHub = N (often equal)

5.2 Index mapping for lacing

Let rim holes be i ∈ [0..N-1].
For each rim hole i, choose hub hole index:

radial: j = i

cross k: j = (i + k*sign) mod N

Where sign alternates to create left/right and leading/trailing symmetry.

A clean deterministic scheme:

Split spokes into 4 groups:
LeftLeading, LeftTrailing, RightLeading, RightTrailing

Each group uses:

flange = left or right

tangential direction = + or −

hub index offset = ±cross

So for each rim index i:

group = i mod 4

side = left if group ∈ {0,1}, right if {2,3}

dirSign = +1 for leading groups, −1 for trailing groups

hubHole = (i + dirSign*cross) mod N

That generates classic 2-cross/3-cross “twist” without hand modeling.

2) Architecture “ANYTHING” — buildings with materials + framing + MEP (electric/plumbing/HVAC)

This is where your system becomes not just a modeler, but a construction engine.

The trick: architecture is naturally multi-layered:

Envelope (walls, floors, roof)

Structure (studs, joists, beams, columns, trusses)

MEP (mechanical/electrical/plumbing: pipes, ducts, conduits)

Finishes (drywall, siding, tile, trim)

Systems metadata (codes, clearances, load paths)

Your engine can represent each layer with the same primitives:

Envelope = SurfaceTemplate (shells + openings)

Structure = GraphTemplate (stud lattice + joist grids + truss graphs)

MEP = GraphTemplate (pipes/ducts/cables along routing graphs)

Finishes = FeatureGraph (trim, reveals, offsets, seams)

Metadata = constraints + hardpoints + “inspection views”

2.1 Architecture Template Families (starter set)
A) HouseTemplate

input: footprint polygon, story heights, roof type

output: wall shells, floor slabs, roof planes, openings

B) FrameTemplate (structure)

studs: vertical array along wall segments

top/bottom plates: sweeps along wall edges

joists: array in floor direction

beams: stronger profiles where spans exceed limits

trusses: graph operator for roof truss layout

C) MEPTemplate

plumbing: pipes with diameter rules + slope constraints (drain lines)

HVAC: duct trunk + branches + registers

electrical: conduit/cable trays + outlets/switches as hardpoints

Each template compiles to geometry + collision + metadata.

2.2 Materials are not “textures”—they’re systems

For architecture you want materials as layer stacks:

Example wall:

exterior siding (render layer)

sheathing (structural)

studs (structural graph)

insulation (volume proxy)

drywall (render layer)

So define MaterialStack:

type MaterialLayer = {
  name: string;
  thicknessM: number;
  kind: "render"|"structural"|"thermal"|"acoustic"|"fire";
  phys?: { density?: number; stiffness?: number; conductivity?: number };
  render?: { materialId: string };
};

type MaterialStack = {
  id: string;
  layers: MaterialLayer[];
};


Then your ThicknessNode can generate correct offsets per layer (or at least per structural vs render).

2.3 Structural realism via constraints (the “codes” layer)

You don’t need full civil engineering to get massive value. You need constraints that prevent impossible layouts:

Structural constraints (starter)

max joist span for given spacing/profile (rule table)

min bearing length on supports

column spacing bounds

truss web angle bounds

minimum thickness for load-bearing walls

MEP constraints (starter)

pipe slope for drainage (e.g. 1–2% grade)

minimum bend radius for conduit/duct

clearance: avoid crossing beams/columns

keep “service zones” (plumbing walls, ceiling plenums)

These become “compile-time validators” that mark issues in the editor.

2.4 The killer UX for architecture

You already have ortho underlays. For buildings, add:

Plan view (top) = blueprint

Elevation views (front/side)

Section cut view (slice plane you can drag)

System toggles: Envelope / Structure / Plumbing / Electrical / HVAC

And your SVG overlay becomes:

draw rectangle = room

draw line = wall segment

draw circle = column

draw spline = duct/pipe route

Each drawing produces nodes/edges in GraphTemplate or wall segments in SurfaceTemplate.

6) How compilation should split into bundles (clean toggles)

Your compiler should emit one AssetBundle with layer manifests:

bundle/envelope/*.glb

bundle/structure/*.glb

bundle/mep/plumbing.glb

bundle/mep/electrical.glb

bundle/mep/hvac.glb

bundle/collision/*.json

bundle/metadata/*.json

bundle/diagnostics/*.json

Then your editor toggles are just enabling groups.

7) The 3 “domain operators” you implement to make this insane
7.1 FrameWallOperator

Input: wall polyline + openings + stud spacing
Output: studs + plates + headers + jacks/king studs

Core rules:

studs at spacing, snapped away from openings

around each opening:

add king studs

add jack studs

add header above

add sill under windows

plates sweep along top/bottom edges

7.2 FloorJoistOperator

Input: footprint + joist direction + spacing
Output: joist edges + rim joists + beams if span exceeds threshold

7.3 MEPRouterOperator

Input: fixtures + service zones + forbidden volumes + constraints
Output: three graphs:

plumbing: supply trunk → branches, drain with slope → stack, vent

electrical: panel → outlets/switches

HVAC: furnace → trunk → branch → registers

V1 can be grid-based + deterministic; still massively useful.
